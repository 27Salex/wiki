Diseño & Normalización

Metodología E/R, Reglas de Transformación y Formas Normales (1FN a 3FN)

Guía Práctica de Normalización: De 1FN a 3FN

La normalización es el proceso sistemático de organizar los datos en una base de datos relacional para evitar la redundancia (duplicación de datos) y prevenir anomalías de inserción, actualización y borrado.

¿Por qué normalizar?
  • Elimina la redundancia física y optimiza el almacenamiento.
  • Garantiza la integridad referencial al evitar desajustes en actualizaciones.
  • Simplifica el mantenimiento del esquema a largo plazo.

Caso Práctico de Partida (Tabla No Normalizada)

Imagina una tabla inicial que almacena información de los artilugios que aparecen en los episodios de una serie:

-- Estructura original con redundancias y valores multivalorados
TABLA Episodios_Artilugios (
    id_episodio        INT,
    titulo_episodio    VARCHAR,
    temporada          INT,
    artilugios         VARCHAR,  -- "Teletransportador, Analizador" (Multivalorado)
    creador_artilugio  VARCHAR   -- "Dr. Baxter, Dra. Vega" (Multivalorado)
);

Primera Forma Normal (1FN): Atomicidad

Una tabla está en 1FN si todos los atributos contienen valores atómicos e indivisibles (sin listas separadas por comas) y existe una Clave Primaria única definida.

-- Paso a 1FN: Desglosamos las listas y definimos una clave compuesta
TABLA Episodios_Artilugios_1FN (
    id_episodio        INT,          -- PK Compuesta (Parte 1)
    titulo_episodio    VARCHAR,
    temporada          INT,
    nombre_artilugio   VARCHAR,      -- PK Compuesta (Parte 2) [Atómico]
    creador_artilugio  VARCHAR       -- [Atómico]
);

Segunda Forma Normal (2FN): Dependencia Completa

Una tabla está en 2FN si ya está en 1FN y todos los atributos que no forman parte de la clave primaria tienen una dependencia funcional completa de toda la clave primaria (es decir, no dependen únicamente de una parte de la clave compuesta).

-- Paso a 2FN: Separamos las entidades según sus dependencias funcionales

CREATE TABLE episodios (
    id_episodio   INT PRIMARY KEY,
    titulo        VARCHAR(150) NOT NULL,
    temporada     INT NOT NULL
);

CREATE TABLE artilugios (
    nombre_artilugio  VARCHAR(100) PRIMARY KEY,
    creador_artilugio VARCHAR(100)
);

CREATE TABLE aparicion_artilugios (
    id_episodio       INT,
    nombre_artilugio  VARCHAR(100),
    CONSTRAINT pk_aparicion PRIMARY KEY (id_episodio, nombre_artilugio)
);

Tercera Forma Normal (3FN): Eliminar Dependencias Transitivas

Una tabla está en 3FN si ya está en 2FN y no existen dependencias transitivas entre los atributos no clave (es decir, ningún atributo no clave puede depender de otro atributo no clave).

-- Paso a 3FN: Eliminamos la transitividad de creador_artilugio -> patrocinador_creador

CREATE TABLE artilugios (
    nombre_artilugio  VARCHAR(100) PRIMARY KEY,
    creador_id        INT
);

CREATE TABLE creadores (
    creador_id        INT PRIMARY KEY,
    nombre_creador    VARCHAR(100) NOT NULL,
    patrocinador      VARCHAR(100)
);

Conversión de Modelo Conceptual (E/R) a Lógico en Relaciones N:M

En el diseño de bases de datos relacionales, una relación Muchos a Muchos (N:M) a nivel conceptual (Diagrama Entidad-Relación) no se puede implementar de forma directa. Debe transformarse obligatoriamente en un modelo lógico mediante la introducción de una tabla intermedia (también llamada tabla de ruptura, unión o pivote).

Reglas de Transformación Estrictas

  1. Cada una de las dos entidades originales se convierte en una tabla independiente con su propia Clave Primaria.
  2. La acción o relación N:M se convierte en una nueva tabla intermedia.
  3. La Clave Primaria de esta nueva tabla es una clave compuesta formada por las claves primarias de las dos entidades relacionadas (que actúan simultáneamente como Claves Foráneas FK).
  4. Cualquier atributo propio de la relación a nivel conceptual se añade como una columna ordinaria en la tabla intermedia.

Caso de Ejemplo Completo: Acción "Sacar"

Imaginemos el siguiente escenario en una serie de ciencia ficción:

  • La entidad Artilugio (identificado por id_artilugio).
  • La entidad Episodio (identificado por id_episodio).
  • La relación N:M "Sacar" (un artilugio sale en muchos episodios, un episodio tiene muchos artilugios).
  • Atributo propio de la relación: minuto (momento del episodio donde se saca el artilugio).
-- 1. Tabla Fuerte: ARTILUGIOS
CREATE TABLE artilugios (
    id_artilugio  INT PRIMARY KEY,
    nombre        VARCHAR(100) NOT NULL,
    descripcion   TEXT
);

-- 2. Tabla Fuerte: EPISODIOS
CREATE TABLE episodios (
    id_episodio   INT PRIMARY KEY,
    titulo        VARCHAR(150) NOT NULL,
    temporada     INT NOT NULL
);

-- 3. Tabla Intermedia: sacar (con PK compuesta y atributo propio)
CREATE TABLE sacar (
    id_artilugio  INT,
    id_episodio   INT,
    minuto        INT NOT NULL,  -- Atributo propio de la relación N:M
    
    -- Clave Primaria Compuesta
    CONSTRAINT pk_sacar PRIMARY KEY (id_artilugio, id_episodio),
    
    -- Claves Foráneas con restricciones de integridad referencial
    CONSTRAINT fk_sacar_artilugio FOREIGN KEY (id_artilugio)
        REFERENCES artilugios(id_artilugio)
        ON DELETE RESTRICT
        ON UPDATE CASCADE,
        
    CONSTRAINT fk_sacar_episodio FOREIGN KEY (id_episodio)
        REFERENCES episodios(id_episodio)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
